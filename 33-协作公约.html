33 · 协作公约

知识库不是「读完就放下的说明书」，而是「25个人共同呼吸的创作大脑」。这份公约定义了怎么用它、怎么养它、怎么让它越滚越厚。

一、知识库的角色定位

这份知识库是活体，不是死文档。

  • 对内：所有人统一的「创作语料库」和「决策参考系」。遇到分歧时，先查知识库里有没有相关讨论（见30-问题与留白）
  • 对外：不公开。内部使用，保护创作先发优势
  • 对AI：如果你的Hermes Agent / Claude / ChatGPT需要上下文，直接从知识库中引用

    • 二、使用规范

    2.1 新成员onboarding

  • 入职第一天：通读本知识库全部33个文件（约2小时）
  • 第一周：带着知识库去完成第一次AI镜头生成实践
  • 第一月：至少对一篇文档提交一条补充/修正

    • 2.2 日常工作引用

    团队讨论和交付物中使用 双链语法引用知识库文件：

  • 编剧讨论剧本 → 「这个场景参考29-意象库第23条」
  • AI组调参 → 「按24-提示词工程第三节海洋模板执行」
  • 后期调色 → 「套27-后期工作流中的妈祖色LUT」

    • 原则：能引用的不重复说，能链接的不复述。

    2.3 版本意识

    每个文件头部有最后更新日期（建议用Git追踪更佳）。引用时注意版本时效性——半年前的提示词模板可能已经过时。

    三、更新机制

    3.1 谁可以更新？

    所有人。 知识库不属于任何一个人。发现错误、有新发现、踩了坑——直接更新。

    3.2 怎么更新？

    
流程：发现问题 → 在对应md文件中补充 → 提交到共享仓库 → 
      在团队群通知「更新了XX文档，主要变化是XXX」
  • 小修正（错别字、链接修正）：直接改
  • 内容补充（新案例、新数据）：追加到对应章节末尾，标注 [更新日期-姓名]
  • 结构调整（拆分/合并章节）：先在群里讨论，2人以上同意后执行
  • 争议内容：不删除原有观点，用「另一种看法」「更新于X年X月」标注并存

    • 3.3 谁来维护？

    指定1名知识库管理员（兼职，建议由编剧组或导演助理兼任）：

  • 每周检查各文件更新情况
  • 整理散落的讨论记录，归入对应文件
  • 每月做一次知识库「体检」：哪些过时了？哪些缺了？

    • 3.4 定期Review节奏

    频率内容负责人
    每周收集本周AI生成参数日志AI组
    每两周回顾[[30-问题与留白]]，补充新问题导演+编剧
    每月全库review，标记过时内容管理员
    每季知识库v2.0大版本更新全员

    四、创作讨论规范

    4.1 讨论前

  • 确认你要讨论的问题是否已在知识库中有记录
  • 带上具体的案例/截图/数据，少说「我觉得」

    • 4.2 讨论中

  • 不说「不行」，说「怎么才行」
  • 技术限制不直接否定创意——先记到30-问题与留白作为待解问题
  • 分歧时回到知识库的基本原则：「以人的故事承载信仰」

    • 4.3 讨论后

  • 结论24小时内落到对应文档
  • 未解决的问题移到30-问题与留白
  • 新的意象/故事种子/术语直接补充到对应文件

    • 五、贡献新发现

    什么值得贡献？

  • AI生成的prompt和参数组合（好的和不好的都要记录）
  • 田野调查中发现的新故事/新细节
  • 看到的好片子、好镜头、好音乐
  • 踩过的坑：浪费了时间的方案、花了钱没效果的尝试
  • 读者/观众反馈：「我妈是信众，她说这一集拍得不对……」

    • 贡献格式

    
在对应文件末尾追加：
---
### [日期] [贡献者姓名] 
[内容]

    例如：在29-意象库末尾添加你发现的新意象。

    六、知识库「体质」指标

    每月检查：

  • [ ] 每个故事种子（28-故事种子）至少有一个被拍出来了吗？
  • [ ] 提示词日志（24-提示词工程第六节）这周更新了吗？
  • [ ] 术语表（31-术语表）有没有新发现的词需要加？
  • [ ] 30-问题与留白有没有新增问题？有没有旧问题被解答？
  • [ ] 这周有没有一个「如果没查知识库就会犯的错」被避免了？

    • 最差的评价：「这个知识库存在，但我们从来不看。」

    最好的评价：「这个知识库，我们每集都在用，而且每集都在往里面加东西。」
